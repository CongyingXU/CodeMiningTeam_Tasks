                "html": "<div id=\"description-val\" class=\"field-ignore-highlight\">\n    <div class=\"user-content-block\">\n                    <p>MDC inheritance when used in conjunction with a thread pool can turn out to be malicious. Consider the following in a web server where each line in the log contains the current user's name.</p>\n\n<p>1. A web request comes in and processing is started on a thread<br/>\n2. Authenticate the user (on original thread)<br/>\n3. Put the user name into the MDC (on original thread)<br/>\n4. Create some Runnable and pass it to a thread pool to be run in the background (on thread pool thread)</p>\n\n<p>Let's take a look at the logs that are printed from the Runnable created in step 4.</p>\n\n<p>Without automatic MDC inheritance (e.g. when using Log4J) these log lines will contain no user name. This can be good or bad, depending on what you want. If you do want the user name in these log lines, you can explicitly transfer the MDC from the originating thread to the thread pool thread. If you don't want to print the user name for background tasks, you are already done. If you have no access to the thread pool or the Runnables, you simply get no MDC information.</p>\n\n<p>However, if you do have automatic MDC inheritance (as with Logback), the log lines <b>will</b> contain a user name, but not from the originating thread. The thread pool thread will instead inherit its MDC from the first thread that executed a Runnable on it. This can never be what you want. You can correct this by transferring the original MDC manually, or clearing it explicitly inside your Runnable. But if you forget to do this, or you don't have access to the thread pool and the Runnables, you will end up with corrupt data in your logs. The latter can happen if you are using some third-party API that maintains its own thread pool.</p>\n\n<p>I can see two ways to fix this issue:<br/>\n1) Make automatic MDC inhertance configurable from logback.xml. This is not a perfect solution: if you mix code that relies on MDC inheritance with code that relies on no MDC inheritance you will have no good value for this setting.<br/>\n2) Remove the automatic MDC inheritance feature altogether. Instead, Logback (or even better: SLF4J) could provide some utility classes to wrap Runnable's and Callable's in something like this:</p>\n\n<div class=\"code panel\" style=\"border-width: 1px;\"><div class=\"codeContent panelContent\">\n<pre class=\"code-java\">class MdcRetainingRunnable <span class=\"code-keyword\">implements</span> <span class=\"code-object\">Runnable</span> {\n  <span class=\"code-keyword\">private</span> <span class=\"code-keyword\">final</span> Map context;\n  <span class=\"code-keyword\">public</span> MdcRetainingRunnable {\n    <span class=\"code-keyword\">this</span>.context = MDC.getCopyOfContextMap();\n  }\n\n  @Override\n  <span class=\"code-keyword\">public</span> void run() {\n    Map originalContext = MDC.getCopyOfContextMap();\n    MDC.setContextMap(context);\n    <span class=\"code-keyword\">try</span> {\n      runWithMdc();\n    } <span class=\"code-keyword\">finally</span> {\n      MDC.setContextMap(originalContext);\n    }\n  }\n\n  <span class=\"code-keyword\">abstract</span> <span class=\"code-keyword\">protected</span> runWithMdc();\n}\n</pre>\n</div></div>\n\n<p>I would prefer solution #2 as it provides for the rule of least astonishment.</p>\n            </div>\n</div>\n",
