                "html": "<div id=\"description-val\" class=\"field-ignore-highlight\">\n    <div class=\"user-content-block\">\n                    <p>This change would introduce incompatibilities with previous versions but would increase general logging performance and decrease size of serialized events.</p>\n\n<p>After I have turned you off sufficiently <img class=\"emoticon\" src=\"/images/icons/emoticons/wink.png\" height=\"16\" width=\"16\" align=\"absmiddle\" alt=\"\" border=\"0\"/>, let me explain my idea:</p>\n\n<p>At the moment, LoggingEvent is handling it's own initialization. It even contains measures against manipulation after initialization in that a call of a setter results in an IllegalStateException if the attribute has already been initialized before.</p>\n\n<p>Instead, I'd suggest that LoggingEvent is changed into a dumb data container with getter and setters for all it's attributes.</p>\n\n<p>If it's really necessary to have a semi-const LoggingEvent I'd suggest to define LoggingEvent as an interface containing only the getters of the attributes and an implementation LoggingEventImpl that implements LoggingEvent including getters and setters.</p>\n\n<p>This communicates the <b>intent</b> that LoggingEvents aren't supposed to be modified but nothing more. It would still be possible to change e.g. the content of an array like argumentArray. Even if the getter would return a copy of the contained array, generating heat in the process, it would <b>still</b> be possible to brutally manipulate the LoggingEvent using java.lang.reflection setAccessible(true) so it's just not possible to guarantee constness.</p>\n\n<p>Instead of having a pseudo-const LoggingEvent let's dump the constness altogether and make it a dumb data container that contains no logic at all. This would instead be moved to the Logger class.</p>\n\n<p>I'd also suggest to remove formattedMessage and change the argumentArray to String[].</p>\n\n<p>This is possible because logback-classic/slf4j promises formatted message, not LoggingEvent.</p>\n\n<p>It would increase performance because a Logger would simply create a LoggingEvent without performing a possibly unnecessary formatting of the message. Creation of LoggingEvents should be as fast and simple as possible, obviously.</p>\n\n<p>The formatting of the message should/would instead be performed in the appender (or, more generally, user of the event) if required.</p>\n\n<p>A serializing appender, an xml appender (e.g. using java.beans.XMLEncoder) or a socket appender would <b>not</b> require a formatted message, executing significantly faster than before.</p>\n\n<p>The argumentArray should be String[] to securely prevent java.io.NotSerializableException in the serializer and . java.lang.ClassNotFoundException in the deserializer.</p>\n\n<p>LoggerRemoteView and LoggerContextRemoteView should be changed the same way as LoggingEvent, i.e. just data container, logic moved to Logger, for similar reasons.</p>\n\n<p>I have to admit, though, that I currently don't understand the reason the LoggingEvent needs to know about LoggerContext beside resolving of the Logger name. I couldn't find code that uses LoggerContextRemoteView.getPropertyMap(), beside loading and saving it.<br/>\nSo I think it may be possible that LoggerRemoteView isn't really necessary and could be removed from LoggingEvent... remembering the Logger name instead.</p>\n\n<p>fqnOfLoggerClass could be omitted because the CallerData would be created in the Logger where fqnOfLoggerClass is known.</p>\n\n<p>And last but not least I'd also change Level into an java.lang.Enum.</p>\n\n<p>I don't expect that all this will be done but I wanted to suggest it nevertheless because it would both increase performance and produce cleaner code.</p>\n\n<p>Breaking compatibility would IMHO be worth it - after all, logback isn't 1.0, yet <img class=\"emoticon\" src=\"/images/icons/emoticons/wink.png\" height=\"16\" width=\"16\" align=\"absmiddle\" alt=\"\" border=\"0\"/></p>\n\n<p>If you are interested I'd volunteer to implement those changes, after signing <a href=\"http://logback.qos.ch/cla.txt\" class=\"external-link\" rel=\"nofollow\">http://logback.qos.ch/cla.txt</a> of course.</p>\n\n<p>Keywords:<br/>\nInversion of Control (IoC), separation of concerns, KISS principle</p>\n\n<p>Related bugs:<br/>\n<a href=\"http://bugzilla.qos.ch/show_bug.cgi?id=100\" class=\"external-link\" rel=\"nofollow\">http://bugzilla.qos.ch/show_bug.cgi?id=100</a><br/>\n<a href=\"http://bugzilla.qos.ch/show_bug.cgi?id=118\" class=\"external-link\" rel=\"nofollow\">http://bugzilla.qos.ch/show_bug.cgi?id=118</a><br/>\n<a href=\"http://bugzilla.slf4j.org/show_bug.cgi?id=70\" class=\"external-link\" rel=\"nofollow\">http://bugzilla.slf4j.org/show_bug.cgi?id=70</a></p>\n            </div>\n</div>\n",
